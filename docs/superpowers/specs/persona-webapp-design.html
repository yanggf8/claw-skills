Scope 定義：

  • ALL — 無過濾，看全部資料
  • OWN — API 強制加 WHERE persona.user_id = :current_user_id（或 user_id = :me），前端看不到其他人
  • NONE — API 回 403，頁面入口也不顯示

Service Token Auth

# claw .env
PERSONA_API_URL=https://api.persona.yourdomain.workers.dev
PERSONA_SERVICE_TOKEN=<secret>

# Skill HTTP header
Authorization: Bearer <PERSONA_SERVICE_TOKEN>
X-Service-Caller: skill/<skill-name>
// Workers middleware
const token = c.req.header('Authorization')?.replace('Bearer ', '')
if (token === env.SERVICE_TOKEN) {
  c.set('user', { role: 'service', sub: null, caller: c.req.header('X-Service-Caller') })
  await next()
  return
}
// 否則走 JWT verify path

role: 'service' 只允許存取上面兩個 write-back 端點，其餘回 403。

Scope 執行位置

// Hono middleware 範例
app.use('/api/personas/:slug/*', async (c, next) => {
  const user = c.get('user')          // JWT payload
  const slug = c.req.param('slug')
  if (user.role !== 'admin') {
    const persona = await db.get('SELECT user_id FROM persona WHERE slug=?', slug)
    if (!persona || persona.user_id !== user.sub) return c.json({ error: 'Forbidden' }, 403)
  }
  await next()
})

audit_log 自動寫入

所有 POST / PUT / DELETE 成功後，middleware 自動插入一筆 audit_log，不需要 route handler 手動呼叫。

Section 4：OAuth 流程 ✅ 確認

sequenceDiagram participant B as Browser (SPA) participant W as CF Workers participant O as OAuth Provider participant DB as Turso B->>W: GET /auth/login/google W-->>B: 302 Redirect → Google consent URL B->>O: User logs in + consents O-->>B: 302 Redirect → /auth/callback?code=xxx B->>W: GET /auth/callback?code=xxx W->>O: POST /token (exchange code → access_token) O-->>W: { access_token, id_token } W->>O: GET /userinfo (email, sub) O-->>W: { email, sub } W->>DB: SELECT * FROM user WHERE oauth_sub=sub alt User exists DB-->>W: user row else First login W->>DB: INSERT INTO user (email, oauth_sub, oauth_provider, role='author') DB-->>W: new user row end W-->>B: Set-Cookie: session=JWT (HttpOnly, Secure, SameSite=Lax)
302 Redirect → / B->>W: API requests with Cookie W->>W: Verify JWT → inject user into context

// Secret: CF Workers Secret (設定於 wrangler secret put JWT_SECRET)

Session 策略

項目決定原因
儲存方式HttpOnly Cookie（非 localStorage）防 XSS 竊取 token
到期7 天夠長不煩人，短到可控
Refresh無 refresh token，到期重新登入MVP 夠用，避免複雜度
Revocation無 server-side blacklistWorkers 無狀態；停用 user 改 DB role，下次 JWT 到期自然失效
Role 變更改 DB 後需重新登入才生效JWT 內含 role，到期前不會更新

First-login 邀請控制

第一次登入自動建 user 有風險（任何人 OAuth 都能進來）。做法：

// /auth/callback 內
if (!existingUser) {
  const invited = await db.get(
    'SELECT 1 FROM invite WHERE email=? AND used_at IS NULL', email
  )
  if (!invited) return c.text('Not invited', 403)
  // 建 user + mark invite used
}

Admin 先在 /api/users/invite 登記 email，對方 OAuth 才能進來。

Section 5：寫作風格預覽機制 ✅ 確認

核心原則

  • CF Workers 不呼叫任何 LLM，純讀 Turso
  • Preview 由 skill（claw / Claude Code）生成，model 與正式發佈一致
  • Webapp 只是讀取並顯示 skill 寫回的結果

Section 6：錯誤處理與測試策略 ✅ 確認

API 錯誤規範

HTTP情境Response body
400Request schema 不合（缺欄位、型別錯）{ "error": "validation", "fields": ["slug"] }
401JWT 缺失或過期{ "error": "unauthorized" }
403Scope 不符（author 存取他人資料）{ "error": "forbidden" }
404slug / id 不存在{ "error": "not_found" }
409slug 重複（create persona）{ "error": "conflict", "field": "slug" }
500DB 連線失敗或未預期錯誤{ "error": "internal" }（不洩漏細節）

Skill 端錯誤策略

  • 延續現有慣例：exit 0，失敗印 [WARN: ...] 不 raise
  • LLM 生成失敗 → preview_text 維持 NULL，下次 cron 重試
  • DB 寫入失敗 → log + skip，不中斷整批
